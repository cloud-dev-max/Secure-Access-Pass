### **Issue #6: Occupancy Math & Summary Cards** 📊
**Priority:** 🔴 Critical  
**Type:** Bug Fix  
**Impact:** High - Accurate occupancy reporting

#### **Problem:**
- Manager summary cards showed stale data from `stats.occupancyBreakdown`
- Did not update dynamically when viewing occupancy tab
- Mismatch between API data and displayed values

#### **Solution:**
- Calculate occupancy cards dynamically from `insideResidents` list
- Resident count = `insideResidents.length`
- Guest count = `insideResidents.reduce((sum, r) => sum + (r.active_guests || 0), 0)`
- Visitor passes = `stats.occupancyBreakdown?.visitor_passes` (from API)
- Total = residents + guests + visitor passes

**Code Changes:**
```typescript
// app/dashboard/page.tsx (Occupancy Tab)
{(() => {
  const residentCount = insideResidents.length
  const guestCount = insideResidents.reduce((sum, r) => sum + (r.active_guests || 0), 0)
  const visitorPasses = stats.occupancyBreakdown?.visitor_passes || 0
  const totalOccupancy = residentCount + guestCount + visitorPasses
  
  return (
    <>
      <div>Total: {totalOccupancy}</div>
      <div>Residents: {residentCount}</div>
      <div>Guests: {guestCount}</div>
      <div>Visitor Passes: {visitorPasses}</div>
    </>
  )
})()}
```

### **Issue #9: Login Reliability** 🔐
**Priority:** 🔴 Critical  
**Type:** Bug Fix  
**Impact:** High - Improves auth reliability

#### **Problem:**
- Login used `.eq('email', email.toLowerCase().trim())` which is case-sensitive in some DBs
- Users with uppercase emails could not log in
- Inconsistent email matching behavior

#### **Solution:**
- Changed from `.eq()` to `.ilike()` for case-insensitive matching
- `.ilike()` is PostgreSQL's case-insensitive LIKE operator
- Ensures reliable email lookups regardless of case

**Code Changes:**
```typescript
// app/api/resident-auth/route.ts
// OLD:
.eq('email', email.toLowerCase().trim())

// NEW:
.ilike('email', email.trim()) // ✅ Case-insensitive
```
